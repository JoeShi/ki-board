# Companion app i18n with react-i18next

The companion desktop app gains English/Simplified Chinese language switching via react-i18next. An i18n module initializes i18next with bundled JSON translation resources, a `LanguageDetector` plugin that checks localStorage then the browser navigator, and English as fallback. All user-facing strings in `main.tsx` are extracted to translation keys, and a two-button language switcher is placed in the sidebar. Language preference persists in localStorage.

**Watch for:** Translation strings containing HTML are rendered via `dangerouslySetInnerHTML` — safe today but a latent XSS surface if translation loading ever changes (confirmed). Pairing event messages are stored as pre-translated strings, so they remain in the old language if the user switches languages after the event fires (confirmed). The language switcher's active-state check uses strict equality against `"en"` / `"zh-CN"`, which may not match what the language detector resolves (likely).

## High-level view

The i18n infrastructure is conventional react-i18next with statically bundled resources, initialized as a module side-effect before React renders. Two strings use `dangerouslySetInnerHTML` because they contain `<strong>` formatting; react-i18next's `<Trans>` component would avoid `innerHTML` entirely and is the library's intended pattern for embedded markup.

Event-driven pairing messages (`pair-event` listener in `useEffect`) call `i18n.t()` at event time and store the result as a plain string in component state. Those messages freeze in whatever language was active when the event fired and won't update on language switch.

The language switcher compares `i18n.language` against the exact strings `"en"` and `"zh-CN"`. When the `LanguageDetector` resolves from `navigator.language`, it may produce region-qualified codes like `"en-US"`. i18next will still find the `en` resource via fallback, but `i18n.language` won't equal `"en"`, so neither button appears active until manually clicked.

## `dangerouslySetInnerHTML` as translation escape hatch

Two translation keys embed HTML formatting: `device.pairHint` (contains `<strong>left + right</strong>`) and `pairModal.matchHint` (contains `<strong>middle (●)</strong>`). Both are rendered via `dangerouslySetInnerHTML={{ __html: t("...") }}`.

The translation files are bundled at build time and team-authored, so there is no active XSS today. The concern is latent: if translations are ever loaded from a server, CMS, or user-contributed source, these two render sites become injection points with no sanitization.

react-i18next provides the `<Trans>` component for this pattern:

```tsx
<Trans i18nKey="device.pairHint" components={{ strong: <strong /> }} />
```

This keeps markup in React's virtual DOM and avoids `innerHTML` entirely.

## Pairing messages frozen in source language

The `useEffect` with an empty dependency array registers a `pair-event` listener once on mount. Inside that listener:

```tsx
case "ok":
  setPairMessage(i18n.t("device.pairSuccess"));
  break;
```

`setPairMessage` stores a rendered string like `"Paired successfully."`. If the user switches to Chinese after this event fires, the rest of the UI re-renders in Chinese but `pairMessage` stays English until the next pairing event.

The fix is to store a key (and optional interpolation params) in state, then translate at render time:

```tsx
// state: { key: "device.pairFailed", params: { reason: "timeout" } }
{pairMessage && <p className="hint">{t(pairMessage.key, pairMessage.params)}</p>}
```

Pairing events are infrequent and the message is ephemeral, so the impact is minor. But the pattern will recur as more event-driven messages are added.

## Language switcher active-state detection

```tsx
className={i18n.language === "en" ? "nav active" : "nav"}
```

`i18n.language` reflects the detected or set language code. When the `LanguageDetector` resolves from `navigator.language`, it may produce `"en-US"` or `"en-GB"`. i18next's resource lookup strips the region and finds the `en` resource, but `i18n.language` remains `"en-US"`.

On first load with a US-English browser and no localStorage value, neither switcher button shows the active state. After the user clicks "EN" once, `changeLanguage("en")` sets the code explicitly and the button highlights correctly from then on.

`i18n.resolvedLanguage` (available since i18next v21; v26 is used here) returns the language key that actually matched a resource — always `"en"` or `"zh-CN"`. Switching to that property fixes the issue.

## Version string duplication

Both `en.json` and `zh-CN.json` contain `"version": "v0.2.0"`. A version bump requires editing both files. Since the string is not locale-sensitive, it belongs as an interpolation variable:

```tsx
{t("sidebar.version", { version: APP_VERSION })}
// translation: "v{{version}}"
```
